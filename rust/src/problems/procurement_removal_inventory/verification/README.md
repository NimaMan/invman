# Verification

`procurement_removal_inventory` is verified by executable assertions in `verification/tests.rs`.

Current verifier scope:

- reference-shape checks from `literature/references.rs`
- policy-state layout checks
- worked-transition accounting checks (purchase -> return-before-liquidate -> lost-sales demand ->
  holding on ending inventory)
- well-formedness of the `REMOVAL_ACTIVE_REFERENCE_INSTANCE`: valid env state, cost ordering matching
  Maggiar & Sadighian (2017) Assumptions 2.ii (`c > s`) and 2.iii (`l <= s`), removal level strictly
  above the order level, overstocked start, and a worked removal step that returns from the
  returnable pool first and liquidates the remainder (Corollary 1)
- exact reduced finite-horizon DP comparison: the optimum dominates both carried heuristics

The reduced exact verifier stores only the problem instance in the literature catalog. Optimal and
heuristic costs are generated by the Rust solver at verification time and are not stored as
literature reference numbers.

## Independent cross-check (2026-05-31 audit)

The Rust exact DP (`finite_horizon_dp.rs`) was re-implemented from scratch in pure Python and run on
`VERIFICATION_PROBLEM_INSTANCE`. The two solvers agree to machine precision:

- Rust exact optimal discounted cost: `31.7802611137`
- independent Python DP: `31.7802611137` (absolute difference `0.00e+00`)

A worked single-step transition was likewise cross-checked. This confirms the env transition + cost +
DP recursion are self-consistent and correctly implemented; it is **not** a literature claim. The
package remains `literature_verified = false` (see `../literature/README.md`).
